# token-saving — Claude Codeのトークン節約リファレンス

Claude Codeは毎メッセージで会話全履歴を読み直す仕組み上、コンテキストが大きいほど1リクエストのコストが増える。
「コンテキストを小さく保つ」ことが節約の核心。以下は公式ドキュメントで裏付けが取れた打ち手（✅）と、
参考程度に留めるべき未確認情報（🔶/⚠️）を分けて整理したチェックリスト。

## 今すぐ効く3手（優先順）

1. **作業の区切りで `/clear`**（バグ修正完了・別機能着手など）。無関係な文脈を持ち越さない。古い会話は`/resume`で呼び戻せる
2. **広範な探索はサブエージェントに委任**。メイン会話には要約だけを残し、探索過程の大量トークンを持ち込まない
3. **CLAUDE.mdは200行以内を目安に維持**し、詳細な手順・リファレンスはSkillへ移す（本ファイルもその実践）

## コマンド操作系

- `/clear`: 空のコンテキストで新規開始。`/resume`で後から呼び戻し可能
- `/compact [指示]`: ここまでの要約に置換。`/compact focus on Xの修正`のように残す観点を指定できる
- `/status` `/context [all]` `/usage`: 役割が違う。`/status`=バージョン確認、`/context`=何が容量を食っているかの診断、`/usage`=コスト会計。節約したいなら`/context`→`/usage`の順で使う
- `/effort`: Sonnet 5・Opus 4.8/4.7は low/medium/high/xhigh/max の5段階。ルーティン作業はlow、難しい設計判断のみhigh以上
- `/model` `opusplan`: planモードはOpus・実行はSonnetに自動切替。ただし切替のたびにプロンプトキャッシュが再構築される（キャッシュミス）点に注意
- `/btw <question>`: 会話履歴に残らない脇道質問。ツールアクセス不可・既存コンテキストのみから回答

## 設定ファイル最適化

- CLAUDE.mdは200行以下。特殊な指示はSkillへ移動しベースコンテキストを小さく保つ
- `permissions.deny`（`.claude/settings.json`）: `Read(./.env)`等で不要な大きいファイルへのアクセスを未然に断つ。評価順は deny→ask→allow
- MCP tool search: ツール名とサーバー指示のみ起動時ロード。実際に使う段でスキーマがロードされるため、MCPサーバー数を増やしてもベースコストへの影響は小さい
- Skillsのprogressive disclosure: 起動時は`name`/`description`のみプリロードされ、本文（SKILL.md）は使われて初めて読み込まれる

## プロセス最適化

- サブエージェント委任: 各サブエージェントは独自のコンテキストウィンドウで動き、要約のみをメインへ返す
- プランモード（`Shift+Tab`循環）: 実装前にレビューすることで手戻り（＝再生成トークン）を防ぐ
- worktree分離: 並列セッションのファイル競合とコンフリクト解消の余分なやり取りを防ぐ
- モデル選択（Sonnetファースト）: 複雑な設計判断・多段階推論が要る場面だけOpusを予約
- 🔶 並列 vs 逐次: 共有コンテキストが要るタスクの並列投げは、各エージェントが同じ前提を個別に読み直し合計トークンが増えることがある（具体的な削減率は非公式）
- 🔶 プロンプトを具体化: 対象ファイル・関数・条件を絞ると探索コストを抑えられる（具体数値は非公式）
- 🔶 1プロジェクト×1スレッド: 無関係なリポジトリを同じセッションで開きっぱなしにしない

## 運用Tips

- 中間ファイル（`.dump.txt`等の作業用ファイル）はタスク完了後に削除し、次回の誤読み込みを防ぐ
- セッション終了時は経緯・決定事項を要点だけメモリ/引き継ぎドキュメントに残す（履歴からの再構築をさせない）
- Skill定義は自分の記憶で代替しない。うろ覚えの手順より都度正しい定義を読み込む方が手戻りが少ない

## よくある誤解（公式ドキュメントで確認できなかったもの）

- ⚠️ `.claudeignore`は公式機能として確認できない。同種の目的を果たす実在機能は`permissions.deny`
- ⚠️ 「Shift+Tab×2でプランモード」という固定回数は公式に明記されていない（モード状態依存）
- ⚠️ `opusplan`の説明として出回る長文引用は公式ページの実際の文言と異なる（実際は「planモードはOpus・実行はSonnet」の一文のみ）
- ⚠️ `/effort`の段階数はモデル依存。Opus 4.6/Sonnet 4.6には`max`はあるが`xhigh`は無い
